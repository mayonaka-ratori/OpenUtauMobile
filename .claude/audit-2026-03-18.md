## Category A: Memory Leaks

### [A-01] Critical — EditViewModel が IDisposable / ICmdSubscriber を実装していない

- **ファイル**: `OpenUtauMobile/ViewModels/EditViewModel.cs`
- **行**: クラス宣言 (~30行目)
- **問題**: EditViewModel は複数のイベント (CollectionChanged: 292, 308, 359行; WhenAnyValue: 282-286行) を購読しているが、IDisposable を実装していない。プロジェクトルール上、すべての ViewModel は IDisposable + ICmdSubscriber の両方を実装し、CompositeDisposable と DocManager.RemoveSubscriber の二重クリーンアップが必要。
- **影響**: EditPage への往復ナビゲーションごとに ViewModel 全体・すべての購読・DocManager への参照がリークする。繰り返しナビゲーションでメモリが際限なく増加。
- **修正案**:
  ```csharp
  // 1. CompositeDisposable フィールドを追加
  private readonly CompositeDisposable _disposables = new();

  // 2. WhenAnyValue 購読に .DisposeWith(_disposables) を追加
  this.WhenAnyValue(x => x.Foo)
      .Subscribe(...)
      .DisposeWith(_disposables);

  // 3. IDisposable.Dispose() を実装
  public void Dispose()
  {
      _disposables.Dispose();
      // CollectionChanged ハンドラも -= で解除
  }
  ```
- **テスト**: EditPage への往復を 20 回繰り返す。`dotnet-gcdump` でマネージドヒープを確認。EditViewModel インスタンス数が 0 か 1 であることを確認。
- **upstream 影響**: なし。モバイル固有 ViewModel。

## Category B: MAUI Lifecycle

### [B-01] High — EditPage に OnDisappearing / OnAppearing ライフサイクル管理がない

- **ファイル**: `OpenUtauMobile/Views/EditPage.xaml.cs`
- **行**: クラス全体
- **問題**: EditPage は `OnDisappearing()` / `OnAppearing()` をオーバーライドしていない。リソースクリーンアップは AttemptExit() から Dispose() を明示呼び出した場合のみ。Android バックボタンや Shell ナビゲーションで AttemptExit を通らずに離脱した場合、Dispose() が呼ばれない。
- **影響**: 明示的な exit フロー外でナビゲーションした場合、タイマー・購読・ジェスチャーハンドラが無期限に残存する。
- **修正案**:
  ```csharp
  protected override void OnDisappearing()
  {
      base.OnDisappearing();
      PlaybackTimer?.Stop();
      AutoSaveTimer?.Stop();
      // 高コスト描画を一時停止
  }

  protected override void OnAppearing()
  {
      base.OnAppearing();
      AutoSaveTimer?.Start();
  }
  ```
- **テスト**: Android バックボタンで EditPage を離脱。タイマーが停止し PaintSurface コールバックが発火しないことを確認。
- **upstream 影響**: なし。

- **テスト**: 再生開始・即一時停止を 100 回繰り返す。常に 100ms 以内に停止することを確認。
- **upstream 影響**: なし。

## Category D: Performance

### [D-01] Critical — EditPage の PaintSurface フレームごとに SKPaint / SKFont を生成

- **ファイル**: `OpenUtauMobile/Views/EditPage.xaml.cs`
- **行**: 1754, 1785-1790, 1949, 2202, 2468-2480
- **問題**: 複数の PaintSurface ハンドラが毎フレーム `new SKPaint()` / `new SKFont()` をアロケートしており、一部は未 Dispose:
  - `PlaybackPosCanvas_PaintSurface` (1754): `using SKPaint paint = new SKPaint()`
  - `PianoKeysCanvas_PaintSurface` (1785-1790): `SKPaint textPaint = new()` + `SKFont font = new()` — **未 Dispose**
  - `PianoRollKeysBackgroundCanvas_PaintSurface` (1949): `SKPaint paint = new SKPaint` — **未 Dispose**
  - `PianoRollPitchCanvas_PaintSurface` (2202): `SKPaint centerPaint = new()` — **未 Dispose**
  - `PhonemeCanvas_PaintSurface` (2468-2480): 複数 SKPaint / SKFont — **未 Dispose**
- **影響**: SKILL Rule 1 違反。毎フレーム 5〜10 以上のネイティブオブジェクト生成。60fps で秒間 300〜600 のネイティブアロケーション → GC 圧迫 → フレームドロップ。未 Dispose のオブジェクトはファイナライザ実行まで漏れ続ける。
- **修正案**:
  ```csharp
  // 変更前: PaintSurface 内
  SKPaint paint = new SKPaint { Color = ..., StrokeWidth = ... };

  // 変更後: static readonly フィールド (または instance フィールド)
  private static readonly SKPaint _playbackPaint = new SKPaint
  {
      Color = SKColors.Red,
      StrokeWidth = 2,
  };
  // PaintSurface 内ではフィールドを直接使用
  ```
- **テスト**: Stopwatch で PaintSurface を計測。8ms 未満であることを確認。修正前後で GC コレクション率を比較。
- **upstream 影響**: なし。

### [D-05] High — GestureProcessor にタッチスロットリングがない

- **ファイル**: `OpenUtauMobile/Views/Utils/GestureProcessor.cs`
- **行**: クラス全体
- **問題**: すべてのタッチイベントが即時処理され、スロットリングがない。SKILL Rule 5 では 60Hz (16ms 最小間隔) が要件。モバイルタッチデジタイザは 120〜240Hz で報告する。
- **影響**: 各タッチイベントが状態更新と場合によっては InvalidateSurface() をトリガーする。240Hz 入力では必要な 4 倍の処理が発生し、CPU 消費増加と InvalidateSurface ストーム (SKILL Rule 6 違反) の可能性。
- **修正案**:
  ```csharp
  private long _lastTouchTicks;

  private bool ShouldThrottle()
  {
      long now = Environment.TickCount64;
      if (now - _lastTouchTicks < 16) return true;
      _lastTouchTicks = now;
      return false;
  }

  // 各タッチハンドラの冒頭で
  if (ShouldThrottle()) return;
  ```
- **テスト**: タッチ〜レンダーレイテンシを計測。32ms 未満であることを確認。高速パン中にフレームドロップがないことを確認。
- **upstream 影響**: なし。

### リスク

1. **SKPaint キャッシュのスレッド安全性**: MAUI では通常 PaintSurface は単一スレッドだが、invalidation storm 時に複数スレッドから呼ばれる可能性は皆無ではない。共有可変 SKPaint 状態には注意。各キャンバスで別インスタンスを使うことで問題を回避できる。
2. **EditViewModel IDisposable と EditPage ライフサイクルの依存関係**: B-01 (OnDisappearing) が修正されていない場合、A-01 の Dispose が適切なタイミングで呼ばれない。B-01 と A-01 を同時に修正することを推奨。
3. **タッチスロットリングのチューニング**: 16ms 閾値がズームなど一部ジェスチャーで応答性を低下させる可能性がある。ジェスチャー種別ごとに閾値を変えることも検討。
4. **Drawable オブジェクトプーリング**: フレームごとの new から長寿命オブジェクトへの変更は、フレーム間での古いデータ描画を防ぐために state management を慎重に設計する必要がある。
